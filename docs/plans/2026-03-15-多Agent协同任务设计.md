# 多 Agent 协同任务设计文档

## 概述

在 GetClawHub Chat 页面实现多 Agent 协同对话功能。通过专用 commander（指挥官）agent 将大任务拆解为子任务，分派给专业 agent 并行执行，最后汇总结果。配合独立的协同任务窗口实时展示执行进度。

## 核心决策

| 决策项 | 方案 | 理由 |
|--------|------|------|
| 协同模式 | 指挥官模式 | commander 拆解任务、分派调度 |
| 指挥官身份 | 专用 commander agent | 和日常聊天职责分离，不污染 main |
| 触发方式 | `/collab` 命令 | 显式触发，用户可控 |
| UI 展示 | 任务卡片 + 独立窗口 | Chat 保留对话，协同窗口展示进度 |
| 调度方式 | App 端调度（Swift 控制） | 完全掌控卡片状态更新 |
| Session 隔离 | `getclawhub-collab-<taskId>-<agentId>` | 每个协同任务、每个 agent 独立隔离 |
| Agent 不足时 | 降级为 main + 角色注入 | 开箱即用，无需提前创建 agent |
| 用户干预 | 自然语言 → commander 返回结构化指令 → App 执行 | AI 理解意图，App 负责执行 |

## 整体架构

### 流程图

```
用户输入: /collab <任务描述>
        │
        ▼
  ┌─────────────────┐
  │  Commander    │  第 1 步：拆解任务
  │  返回 JSON       │
  └────────┬────────┘
           │ App 解析 JSON，渲染任务卡片
           │ 自动弹出协同窗口
           ▼
  ┌────┬────┬────┐
  │ 子任务1 │ 子任务2 │ 子任务3 │   任务卡片（全部"待处理"）
  └──┬─┘└──┬─┘└──┬─┘
     │     │     │
     ▼     ▼     ▼           第 2 步：执行子任务
  App 调用: openclaw agent --agent <agentId>
            --session-id getclawhub-collab-<taskId>-<agentId>
            -m '<子任务提示词>'
  卡片状态实时更新
           │
           ▼ 全部完成
  ┌─────────────────┐
  │  Commander    │  第 3 步：汇总结果
  │  最终报告         │
  └────────┬────────┘
           │
           ▼
     最终结果展示在 Chat 页面
```

### 三轮 API 调用

1. **拆解** — 将任务发给 commander，拿回子任务 JSON
2. **执行** — App 按依赖关系逐个/并行调用子 agent，收集结果
3. **汇总** — 将所有子任务结果喂回 commander，生成最终回复

## Commander Agent

### 角色定义

Commander 是一个专用 agent，在 `~/.openclaw/openclaw.json` 的 `agents.list` 中创建。

**文件约定（遵循 openclaw 规范）：**
- `IDENTITY.md` — 结构化身份信息（name/emoji/creature/vibe），写入 **workspace** 目录
- `SOUL.md` — 行为指导 persona，写入 **workspace** 目录（openclaw 从 workspace 读取，不是 agentDir）
- workspace 路径：`~/.openclaw/workspace-commander/`
- agentDir 路径：`~/.openclaw/agents/commander/agent`

Commander 的 persona 专门优化为：

- 理解用户任务需求
- 将大任务拆解为可执行的子任务
- 根据能力描述匹配合适的 agent
- 判断何时拆分、何时合并（任务拆分决策规则）
- 输出结构化 JSON
- 汇总多个 agent 的执行结果

### Agent 能力描述注入

App 在调用 Commander 拆解任务时，会从每个 agent 的 `IDENTITY.md`（`---` 分隔符后的描述文本）和 `SOUL.md`（`## You Are` 段落）中提取能力描述，注入到 `[Available Agents]` 中。这使 Commander 能够基于 agent 的实际能力做出更准确的任务分配。

示例：
```
[Available Agents]
- coder: coder — 擅长编写代码、调试、重构，支持 Python/Swift/JS 等主流语言
- reviewer: reviewer — 专注代码审查、质量分析、安全漏洞检测
```

### 输入格式（拆解阶段）

App 在调用 commander 时，自动注入可用 agent 列表和用户任务：

```
[Available Agents]
- coder: 擅长编写代码、调试、重构
- reviewer: 擅长代码审查、质量分析
- writer: 擅长文档编写、内容创作

[User Task]
帮我开发一个 TODO 应用，要有完整的代码和文档

请将此任务拆解为子任务并分配给合适的 agent。
以如下 JSON 格式返回：
{
  "summary": "计划简述",
  "tasks": [
    {
      "id": 1,
      "title": "子任务标题",
      "agent": "agent_id 或 null（无匹配时）",
      "role": "降级角色描述（agent 为 null 时使用）",
      "prompt": "给子 agent 的详细提示词",
      "depends_on": []
    }
  ]
}
```

### 输出格式（拆解阶段）

```json
{
  "summary": "开发 TODO 应用，分 3 个子任务",
  "tasks": [
    {
      "id": 1,
      "title": "编写核心代码",
      "agent": "coder",
      "role": null,
      "prompt": "请用 Swift 开发一个 TODO 应用，支持添加、删除、完成切换...",
      "depends_on": []
    },
    {
      "id": 2,
      "title": "审查代码质量",
      "agent": "reviewer",
      "role": null,
      "prompt": "请审查以下 TODO 应用代码，检查代码规范、潜在 bug...",
      "depends_on": [1]
    },
    {
      "id": 3,
      "title": "编写使用文档",
      "agent": null,
      "role": "技术文档工程师",
      "prompt": "请为这个 TODO 应用编写用户文档...",
      "depends_on": [1]
    }
  ]
}
```

关键字段说明：
- `agent`：匹配已有 agent 的 ID；无合适 agent 时为 `null`
- `role`：降级角色描述，`agent` 为 `null` 时使用（main agent + 角色注入）
- `depends_on`：前置依赖的任务 ID 列表；为空表示可立即执行

### 执行逻辑

```
解析 JSON 中的 tasks
构建依赖关系图

while (存在未完成的任务):
    找出所有 depends_on 已全部完成的任务
    并行执行这些任务:
        if task.agent 存在且可用:
            openclaw agent --agent <task.agent>
                           --session-id getclawhub-collab-<taskId>-<task.agent>
                           -m '<task.prompt>'
        else:
            openclaw agent --agent main
                           --session-id getclawhub-collab-<taskId>-main
                           -m '[Role: <task.role>] <task.prompt>'
    收集结果，更新卡片状态
    将已完成任务的结果注入到后续依赖任务的 prompt 中

将所有结果喂回 commander 生成最终汇总
```

### 降级策略

优先级：**专属 agent > main + 角色注入**

当 `task.agent` 为 `null` 或指定的 agent 在系统中不存在时：
- 使用 `main` agent，在 prompt 中注入角色描述
- 示例：`-m "[Role: 技术文档工程师] 请为这个 TODO 应用编写用户文档..."`
- 这保证了即使用户没有创建任何自定义 agent，`/collab` 也能直接使用

## 任务拆分决策规则

Commander 在拆解任务时遵循以下决策规则：

**核心原则：拆分是为了并行提效，不是为了拆而拆。高耦合任务强行拆分反而增加沟通成本。**

### 适合拆分的场景（拆）
- 子任务之间**逻辑独立**，可以并行执行（例如：前端和后端分别开发）
- 子任务需要**不同专业能力**，有专属 Agent 更合适（例如：代码编写 vs 文档撰写）
- 子任务之间只需要**单向传递结果**（例如：先写代码，再做审查，结果传递即可）

### 不适合拆分的场景（合）
- 子任务之间需要**频繁共享上下文**（例如：同时修改多个紧密关联的文件）
- 子任务涉及**同一模块的增量修改**（例如：先写基础功能，再在同一文件加高级功能）
- 拆分后各子任务都由**同一个 Agent** 执行，且是**串行依赖**的（既然都给同一个人做，拆开反而丢失上下文）
- 子任务的输出是另一个子任务的**输入的核心部分**，且难以通过简单文本传递

### 判断流程
1. 先识别任务中有哪些逻辑独立的模块
2. 对每个模块，判断它应该由哪个 Agent 执行
3. 如果连续多个模块都归同一个 Agent 且是串行依赖，**合并为一个子任务**，在 prompt 中说清楚分步做什么
4. 只有真正可并行、或由不同 Agent 执行的模块才拆分为独立子任务

## Session 隔离与合并

### Session ID 命名规则

```
日常聊天:        --session-id getclawhub-chat-<agentId>
帮助助手:        --session-id getclawhub-help-assistant
协同任务:        --session-id getclawhub-collab-<collabTaskId>-<agentId>-<subtaskId>
```

- `collabTaskId`：每次 `/collab` 生成的 UUID
- 协同任务中每个子任务默认有独立 session
- 协同 session 不会污染日常聊天和帮助助手

### 同 Agent 串行任务的 Session 合并

当满足以下条件时，后续子任务会**复用前驱任务的 session ID**，从而保留完整对话上下文：

1. 当前子任务只有 **1 个直接依赖**（`depends_on` 数组长度为 1）
2. 前驱子任务使用的是**同一个 agent**
3. 两��子任务是**串行关系**

**效果**：同 agent 的串行链条（如任务 1 → 2 → 3 都给 coder）会共享同一个 session，agent 可以看到之前的完整执行历史，无需通过文本传递丢失上下文。

**示例**：
```
任务 1 (coder): session = getclawhub-collab-abc-coder-1
任务 2 (coder, depends_on: [1]): session = getclawhub-collab-abc-coder-1  ← 复用
任务 3 (reviewer, depends_on: [2]): session = getclawhub-collab-abc-reviewer-3  ← 不同 agent，新 session
```

## 双窗口设计

### Chat 页面（现有）

- 用户输入 `/collab <任务描述>` 触发协同任务
- Commander 在 Chat 中回复："已拆解为 N 个子任务，正在执行"
- 协同窗口自动弹出
- 用户可以继续在 Chat 中与 commander 对话：
  - 查询进度："现在进展怎么样？"
  - 下达指令："跳过任务 3"、"重试任务 2"
  - 修改需求："任务 1 的代码加个截止日期功能"
  - 取消任务："取消整个任务"

### 协同任务窗口（新增，独立窗口）

参照 HelpAssistantWindow 的模式创建独立窗口。展示内容：

#### 任务卡片状态

| 状态 | 图标 | 颜色 | 说明 |
|------|------|------|------|
| 待处理 | ⏳ | 灰色 | 等待前置依赖完成 |
| 进行中 | 🔄 | 蓝色 | 子 agent 正在执行 |
| 已完成 | ✅ | 绿色 | 已收到执行结果 |
| 失败 | ❌ | 红色 | 执行出错，可重试 |
| 已跳过 | ⏭️ | 灰色 | 用户选择跳过 |

#### 展开卡片内容

点击"展开"后显示：
- 子 agent 的完整回复（Markdown 渲染）
- 失败时：错误信息 + [重试] 按钮
- 执行耗时

## Chat 与协同窗口联动

### 进度查询

用户在 Chat 中问进度时，App 自动将当前任务状态注入上下文：

```
[Task Status]
#1 编写核心代码 — ✅ 已完成（耗时 12 秒）
#2 审查代码质量 — 🔄 进行中（已耗时 8 秒）
#3 编写使用文档 — ⏳ 等待中

[User Question]
现在进展怎么样？
```

Commander 根据注入的状态自然回复。

### 用户指令处理

用户在 Chat 中用自然语言下达指令 → commander 返回结构化 JSON → App 解析执行。

#### Commander 响应格式

**普通回复（仅回答问题）：**
```json
{
  "type": "reply",
  "message": "任务 1 已完成，任务 2 正在审查中，整体进度 1/3。"
}
```

**操作指令：**
```json
{
  "type": "action",
  "action": "skip",
  "taskId": 3,
  "message": "好的，已跳过任务 3"
}
```

#### 支持的操作类型

| 操作 | 说明 | 附加字段 |
|------|------|----------|
| `skip` | 跳过指定任务 | `taskId` |
| `retry` | 重试指定任务 | `taskId` |
| `cancel_all` | 取消整个协同任务 | — |
| `modify` | 修改任务描述后重新执行 | `taskId`、`newPrompt` |

### 修改任务流程

当用户说"任务 1 的代码加个截止日期功能"时：
1. Commander 返回 `{"type": "action", "action": "modify", "taskId": 1, "newPrompt": "..."}`
2. App 用新 prompt 重新执行任务 1
3. 所有 `depends_on: [1]` 的后续任务也随之重新执行

## JSON 解析容错

由于 LLM（如 qwen3.5-plus）可能不严格遵循 SOUL.md 中的字段名约束，App 端实现了 `normalizeDecomposeJSON` 兜底解析器：

- 顶层 key：`summary` / `task` / `plan` / `overview` / `description` → `summary`
- 任务数组：`tasks` / `subtasks` / `sub_tasks` / `steps` → `tasks`
- 任务字段：`name` → `title`，`description` → `prompt`，`assignedTo` / `assigned_to` / `agentId` → `agent`，`dependsOn` / `dependencies` → `depends_on`

解析流程：先尝试标准 `JSONDecoder`，失败后走 `normalizeDecomposeJSON` 兜底。

## 流式拆解进度

Commander 拆解任务期间使用 `runCommandStreaming` 替代 `runCommand`，在协同窗口实时显示 Commander 的思考过程（streaming output），避免用户在拆解阶段看到"卡住"的假象。

相关 Published 属性：
- `isDecomposing: Bool` — 是否正在拆解
- `decomposingOutput: String` — 拆解过程中的流式输出片段

## 需要新增/修改的文件

### Swift 数据模型

```swift
// 子任务状态
enum CollabTaskStatus {
    case pending          // 待处理
    case inProgress       // 进行中
    case completed        // 已完成
    case failed(String)   // 失败（附错误信息）
    case skipped          // 已跳过
}

// 单个子任务
struct CollabSubTask: Identifiable {
    let id: Int
    let title: String
    let agentId: String?       // nil = 使用 main + 角色注入
    let role: String?          // 降级角色描述
    let prompt: String
    let dependsOn: [Int]
    var status: CollabTaskStatus = .pending
    var result: String?        // agent 返回的结果
    var elapsedTime: TimeInterval?
}

// 协同任务会话
struct CollabSession: Identifiable {
    let id: String             // UUID
    let taskDescription: String
    var summary: String        // commander 的计划摘要
    var subtasks: [CollabSubTask]
    var finalResult: String?   // 最终汇总结果
    let createdAt: Date
}

// 协同任务 ViewModel
@MainActor
class CollabViewModel: ObservableObject {
    @Published var session: CollabSession?
    @Published var isRunning = false
    @Published var isDecomposing = false       // 拆解阶段进行中
    @Published var decomposingOutput = ""      // 流式拆解输出

    private weak var dashboardViewModel: DashboardViewModel?

    func startCollab(_ taskDescription: String) async { ... }
    func skipTask(_ taskId: Int) { ... }
    func retryTask(_ taskId: Int) async { ... }
    func cancelAll() { ... }
    func modifyTask(_ taskId: Int, newPrompt: String) async { ... }
}
```

### Commander Agent 配置

App 首次启动或初始化时：

1. 在 `~/.openclaw/openclaw.json` 中创建 commander agent 条目：
```json
{
  "agents": {
    "list": [
      {
        "id": "commander",
        "name": "commander",
        "default": false,
        "identity": { "name": "Commander", "emoji": "🎯" },
        "agentDir": "~/.openclaw/agents/commander/agent",
        "workspace": "~/.openclaw/workspace-commander"
      }
    ]
  }
}
```

2. 在 workspace 目录 `~/.openclaw/workspace-commander/` 写入：
   - `IDENTITY.md` — Commander 身份信息
   - `SOUL.md` — Commander 行为指导（从 app 内置 `commander-persona.md` 加载）

每次 `ensureCommanderAgent()` 都会更新 SOUL.md，保证 persona 与 app 版本同步。
