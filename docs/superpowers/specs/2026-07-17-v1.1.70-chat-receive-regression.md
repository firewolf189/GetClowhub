# v1.1.70「消息发送后收不到回复」根因分析与修复(实证定案)

- 日期:2026-07-17(晚间修订:**推翻早前的 runId 假说,以真机实证定案**)
- 症状:v1.1.70 发出消息后,助手回复永不显示;v1.1.69/v1.1.71 正常
- 状态:**已修复并真机 E2E 验证**(分支 `fix/chat-run-receive`)

---

## 一句话真因(实证)

**openclaw 网关把 sessionKey 规范化为小写**并盖在每条 chat 事件上,而客户端用 `UUID().uuidString`(大写)派生 sessionKey。v1.1.70 的重构在**事件接收路径新增了大小写敏感的 sessionKey 全等门**(hub `matches()` + 消费循环 guard)→ `agent:main:CF0E…`(订阅) ≠ `agent:main:cf0e…`(事件)→ **每条回复在送达消费者前被静默丢弃**。

### 真机诊断日志铁证(2026-07-17 19:25,公开值诊断构建)

```
发送: sessionKey=agent:main:CF0E0CB2-F6B1-4EB7-84FB-0301660725A4   (大写)
事件: sessionKey=agent:main:cf0e0cb2-f6b1-4eb7-84fb-0301660725a4   (小写!)
runId: 5C2B438D-…(ack)== 5C2B438D-…(事件)                        (runId 完全一致!)
结果: hub.matches() sessionKey 门失败 → 丢弃;消费者从无 phase=chat_first_event
```

**runId 从头到尾不是问题**:网关在 chat.send 的 res 里返回 runId,事件原样携带同一 runId。

## 为什么 v1.1.69/v1.1.71 正常

v1.1.69 主聊天消费循环的守卫**只有 `eventRunId == runId`,不检查 sessionKey**(ChatHelpers L685-827),且 hub 广播全量不过滤 → 大小写差异从未被比较过。(注:v1.1.69 的 image-review 小循环有 sessionKey 守卫,推测该冷门路径当时也是坏的,无人发现。)

## 修复(分支 fix/chat-run-receive)

1. **单点根治**:`sessionKeyForAgent`(全 app 唯一 sessionKey 构造点,ChatHelpers.swift:23)派生结果 `.lowercased()`——客户端从源头说网关的规范形,发送/订阅/守卫/RPC 全链一致。
2. **纵深防御**:hub `matches()` 的 sessionKey 比较改为 `caseInsensitiveCompare`,保护遗留大写键的订阅(如旧的 in-flight 恢复记录)。
3. **保留第一轮的 hub provisional 修复**(见下),它防护的是「网关 runId 尚未绑定」的窗口期,与本修复正交、无害且有单测。

### 真机验证(2026-07-17 19:29)

发送 `7+7=?只回复数字` → 6 秒后 **"14" 渲染**;诊断链完整:`chat_send_start(小写 key)→ hub.match 命中 → phase=chat_first_event/first_delta(消费者收到,此前所有失败 run 从未出现)→ final 渲染`。单测含大小写回归断言(大写订阅 × 小写事件必须投递)。

## 附:第一轮修复(runId 假说)的价值与教训

早前静态分析得出「订阅用 idempotencyKey 而网关分配自己的 runId → hub runId 门丢事件」的结论并做了 provisional-routing 修复。**实证表明该假说不成立**(网关 res 的 runId 会回传且事件一致携带)。教训:

- **纯代码推演在双端协议问题上会骗人**——网关"应该"怎么发不等于实际怎么发。定案必须拿到**线上两侧的真实值**(本案靠 Debug 构建把 os.log 私有值改 `privacy: .public` 才抓到)。
- os.log 默认把插值 redact 成 `<private>`,排障时按需加 `privacy: .public`(本次已把 final 事件的 runId/sessionKey 保留为公开,delta 降回 debug 级)。
- provisional 修复保留:它对「事件先于 ack 到达」的真实窗口仍有意义,且 hub 单测证明其行为正确。

## 排障方法存档(可复用)

1. `log stream --process GetClawHub --info --debug` 后台抓 → 驱动真机发消息 → 看 `phase=chat_send_*` / `chat event:` / `phase=chat_first_*` 断在哪一环。
2. 值被 `<private>` 遮蔽时:Debug 构建把关键日志加 `privacy: .public`(或临时加逐订阅匹配诊断日志),重建重测,几分钟拿到铁证。
3. 网关摘要日志 `~/Library/Logs/openclaw/gateway.log` 的 `res ✓ chat.send` 行是**响应完成**才记录——网关楔死时收了请求不记日志,勿据此断言"请求没到"。
4. 网关有 chat-turn 楔死发作期(上游 bug,任何客户端版本都拿不到回复);对比测试必须先用已知正常版确认网关健康,否则结论无效。
